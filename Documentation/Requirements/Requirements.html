(1.0) Introduction

(1.1) Purpose

The purpose of this project is to provide Pokémon players with a comprehensive database of relevant information which will allow them to make educated decisions on how they would like to play the game. Specifically, how they can optimize their Pokémon teams based on information our application provides.

(1.2) Scope

We aim to have all 151 Generation I Pokémon in our database, along with the following information which has already been provided by the ingame Pokédex: name, description, type(s), height, weight, brief biography description, and location. As well as additional information that is relevant but not provided in-game: abilities, catch rate, type match ups, leveling-rate, and EV yield. We would also like to have a user account system using a password hashing algorithm to store passwords. Users will be able to save information to their account.

(1.3) Definitions, acronyms, and abbreviations

Move: A move, also known as an attack is the skill Pokémon primarily use in battle.

Type: Pokémon and the different moves they can use are all classified by properties called Types. A Pokémon can be associated with one or two types whereas a move can only be associated with one type. Moves that deal damage to other Pokémon can have more of an effect, a neutral level of effectiveness, less of an effect, or no effect at all, based on their type and the type of the Pokémon they are being used against. An example of an effective move would be a Pokémon using a Fire type move against a Grass type Pokémon. An example of a less effective move would be a Pokémon using a Fire type move against a Water type Pokémon. Finally, an example of a move which would have no effect at all would be a Ghost type move being used against a Normal type Pokémon.

Abilities: An Ability is a game mechanic that provides a passive effect in battle or in the overworld. Individual Pokémon may have only one Ability at a time. Not every Ability is entirely beneficial; some of them are hindering.

Catch rate: Each species of Pokémon has a catch rate that applies to all its members. When a Poké Ball is thrown at a wild Pokémon, the game uses that Pokémon's catch rate in a formula to determine the chances of catching that Pokémon. Higher catch rates mean that the Pokémon is easier to catch, up to a maximum of 255.

Type matchup chart: A type chart, also known as type matchup chart, shows which modifiers are applied to move types when attacking Pokémon of each type.

EV yield: When a Pokémon is defeated in battle, it will give effort values to the Pokémon that participated in the battle against it.

(1.5) Overview

The Pokédex Encyclopedia(“the project”, “the software”) is a third party database created for the access of information on Generation I Pokémon. The purpose of this application is to give Pokémon players access to a greater depth of information on Pokémon that is not provided by the in-game Pokédex. This encyclopedia will be a guide for players who aims for completion rather than the be all and end all. Vital information like base stats, in-game locations, type effectiveness, natures, and learnset is not provided by other online encyclopedias. The Pokédex Encyclopedia strives to provide this information to allow the players to make educated decisions on how they would like to play the game.

(2.0) Overall description

(2.1) Product perspective

The Pokédex Encyclopedia is a desktop program that provides an easily accessible database for information on all Pokémon in generation I. This will allow users to make educated and informed decisions on how they would like to play the game.

(2.2) Product Functions

  • Look up specific Pokémon and return their stats, moves, type, etc.
  • Side by side compare Pokémon stats with other Pokémon
  • Calculate damage against other Pokémon
  • List where specific Pokémon can be found in-game
  • Have a user system where users can save information

(2.3) User Characteristics

The intended users of our product will be all users of all ages and of all technical experience that are interested in Pokémon

(2.4) Constraints

There will be no cost constraints. The only constraints that we predict to encounter will be development constraints.These include time and possible reliability constraints depending on which apis or frameworks we utilize. Furthermore, team members will be expected to learn how to use the software or any additional knowledge that they may need to complete the tasks they are responsible for.

(3.0) Requirements

(3.1) External interface requirements

(3.1.1) User interfaces

The user will have, at a minimum, the following six interfaces to navigate the project:
Log-in Interface To access the project, the user must log-in. This will be the initial web page that they will interact with. It will have a menu, centred in the screen, in which they can type in their username first and then their password, just below, to log into their account. If the log-in is successful, proceed to Interface #2 (Homepage interface). Otherwise, provide an error message and prompt the user to either provide different log-in credentials.
Homepage Interface The homepage interface will follow a successful log-in. From here, the user will have four options (buttons) to choose: the search function, their saved Pokemon list, their Teams list, and a side-by-side comparison tool.Furthermore, at the top left of each page/interface, there will be a button that can return the user to the homepage.
Search Interface The search function will have one search bar on the page that will prompt the user to type in the name of the Pokemon that they want to search.Before any search has been completed, the full list of Pokemon will be displayed. It can be filtered by any statistic using a filter menu contained on the screen.After being searched for, the interface will return the Pokemon that had been typed in. The user can select and confirm that Pokemon, which will move the user to Interface #4, or search again.
Pokemon Profile Interface The profile will show all information in the database related to the chosen Pokemon in a table.The table will consist of: Name, HP, Attack, Defense, Special Attack, Special Defense, Speed, Evolve Level, Location(s), Type.There will be a button for the user to save the Pokemon to their ‘Saved Pokemon’.
Saved Pokemon Interface The interface will show a list of Pokemon, ordered alphabetically, that the user has chosen to save.If a Pokemon is selected, it will show the user Interface #4 (Pokemon Profile) for that respective Pokemon.
Comparison interface The user will select a Pokemon, if one is not selected already, from a list. The picture and table, a smaller version of Interface #4, will appear on the left half of the screen. From there, they can search for another Pokemon and add them to the interface. Once this is completed, the picture and table for the second Pokemon will appear on the right hand side.T here will be an option to switch out one of the Pokemon and search for another using a search bar below both statistics tables.
Team List Interface A list with 3 spots.Each spot will have the name of a team that has been created by the user or ‘Empty’ where there are available spots. If the user selects a team, they can view the team and modify it in Interface #8.
Team Selection Interface There will be 6 slots available in which a Pokemon can be selected. There will be two rows of three slots. The user can select a Pokemon from the list to place in each spot. There will be a save button in the top right hand corner of the page which will allow the user to name and save the team so that it can be accessed in the future from Interface #7. Next to the save button will be an option to clear the data for the team (Pokemon chosen and team name) so that the user can start a new team.
The aspects of optimizing the interface are: The error messages should be under 75 words. All labels in buttons should be under 10 words. Individual tables should have two columns, the first will identify which statistic/attribute the row is for and the second holds the value(s), and 10 rows, one for each of the ten values (Name, HP, Attack, Defense, Special Attack, Special Defense, Speed, Evolve Level, Location(s), Type). Tables used in comparison should have three columns--one to identify which attribute or statistic is in the row, one for the value for Pokemon 1, and the last for the value for Pokemon 2. Each cell should have less than 10 words. The button to return to the homepage should be in the left-side of the header for all other web pages.

(3.1.2) Hardware interfaces

This software is meant to be a web-based application. So the hardware requirements for it would include a working phone or computer (with mouse and keyboard). This means that mac or windows, iPhone or android, the website should work for all of them. That being said the optimization for mobile devices will not be ideal.

(3.1.3) Software interfaces

The web-application requires a web browser to be run. In addition to this the web-application will need to access a database we have created.

(3.1.4) Communications interfaces

The web-application as well as the database will be hosted on laurier servers.

(3.2) Functional requirements

Searching Pokemon

Introduction/Purpose of feature This is the core functionality of the application. This website gives the user the ability to search through generation 1 pokemon and easily find all of their statistics and other important information.
Stimulus/Response sequence There will be a website fixing on a monthly basis, and a moderator will have authority and the access to check if there is any need to update, delete or edit the website.
Associated functional requirements The user should be able to easily find the search bar and type in the name of a pokemon they are looking for and have it come up. Additionally, we will make it so that there is partial autofill for searches, so "char" would give "charmander" and "charmeleon" and "charizard".

Creating Pokemon team

Introduction/Purpose of feature This is a quality-of-life feature for the user. If an account is made they can save pokemon to a team so that when they come back in future and are looking for the same group of pokemon that information is easily available to them.
Stimulus/Response sequence There will be a website fixing on a monthly basis, and a moderator will have authority and the access to check if there is any need to update, delete or edit the website.
Associated functional requirements The user should be able to create a team of pokemon by going to a specific part of the web application and searching through the list of generation 1 pokemon and adding up to 6 pokemon per team. With a maximum of 3 teams per account.

Comparing Strengths and Weaknesses of Pokemon

Introduction/Purpose of feature This feature allows the user to compare strengths and weaknesses of pokemon. This is very useful because when fighting in teams it is very helpful to know which are stronger and weaker against each other. This feature helps the user get an advantage when fighting against another team of pokemon.
Stimulus/Response sequence There will be a website fixing on a monthly basis, and a moderator will have authority and the access to check if there is any need to update, delete or edit the website.
Associated functional requirements When looking through the different pokemon available you should be able to click on two different pokemon and click compare to see their strengths and weaknesses.

The feature would display if the chosen pokemon counters the enemy or not. If not, it can recommend the types of pokemon in the team who counters the ene

Grouping by Variable

Introduction/Purpose of feature This helps the user look through pokemon grouped by type, specific stat, or location. This can be very helpful when trying to create a well rounded pokemon team.
Stimulus/Response sequence There will be a website fixing on a monthly basis, and a moderator will have authority and the access to check if there is any need to update, delete or edit the website.
Associated functional requirements A group option should be available at the top of the list of pokemon allowing the user to group by type, weakness, strength, location.

(3.3) Performance requirements

  • The web-application should support as many users as the Laurier servers will allow.
  • It should not take more than 2 seconds for a pokemon search to give results.
  • It should not take more than 2 seconds to load a pokemon team that the user created.
  • It should not take more than 2 seconds to load strength vs weakness statistics between pokemon
  • The database should be able to store 10,000 users with each user being able to create 3 teams

(3.4.2) Security

When storing user passwords we will implement a hashing algorithm to keep their passwords as secure as possible. We will keep a log of the when and where users logged in from to try and better identify any possible attacks on the website. We will use encapsulation whenever possible to try and keep the various methods and data stored as safe as possible. Some of the techniques wer plan on use are the following:

  1. Utilize certain cryptographical techniques
  2. Keep specific log or history data sets
  3. Assign certain functions to different modules
  4. Restrict communications between some areas of the program;
  5. Check data integrity for critical variables.

(3.4.3) Maintainability

This web-application should be easily updatable because all that you would need to update is the web-application files where the server is hosted and it will be seamlessly updated for all users once they refresh their web pages. In addition, code needing to be updated should be easy to update through good coding practices like documentation and modularity of the various methods and functions.

(3.4.4) Portability

From a portability perspective this web-application should be available on any device with no setup required. By nature it should be able to be viewed and used on any system with internet and access to a web browser.
